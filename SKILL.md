# dig-pain：评论区痛点矿工

## 目标
从 AI 行业帖子（X/Twitter、YouTube）的评论区中挖掘用户痛点，输出结构化的中文分析报告。

## 执行流程

### 阶段 1：获取当日帖子源并生成中文版 Markdown（必须先完成）

1. **调用 `follow-builders` 技能获取当日数据**
   - 使用 `Skill` 工具调用 `follow-builders`，**限制条件为只获取今日数据**。
   - **关键要求**：将获取到的内容（英文推文、播客等）翻译成**中文**，并为每个帖子生成摘要。
   - 保存为 Markdown 文件：`follow-builders-<YYYY-MM-DD>.md`

2. **调用 `aihot` 技能获取当日数据**
   - 使用 `Skill` 工具调用 `aihot`，**限制条件为只获取今日数据**。
   - 将获取到的 AI 热点资讯整理为**中文版**，每个条目包含摘要。
   - 保存为 Markdown 文件：`aihot-<YYYY-MM-DD>.md`

3. **Markdown 文档格式要求（必须遵守）**

   每个帖子/条目必须包含：
   ```markdown
   ### [序号]. [帖子标题/主题]
   - **作者/来源**：<姓名> (@<handle>) / <公司/角色>
   - **发布时间**：<日期>
   - **链接**：<URL>
   - **摘要**：<100-200字中文摘要，概括核心观点>
   ```
   **注意**：不需要输出互动数据。

4. **链接校验（关键卡点）**
   - 从上述两个 Markdown 文件中提取所有帖子链接（URL）。
   - **如果未提取到任何链接**，立即停止后续流程，并回复用户：
     > "未从 follow-builders 和 aihot 的输出中找到帖子链接。请确认数据源中包含有效的帖子链接。"
   - **只有当至少拿到一个链接时**，才进入阶段 2。

### 阶段 2：抓取所有链接的评论并筛选（使用 Chrome DevTools MCP）

**重要原则**：必须遍历完所有链接，不能中途停止。对每个链接，打开页面后**先判断帖子发布日期是否为近两日（今日或昨日）**（只有今日或昨日发布的帖子才抓取），**再判断评论数**（≥10条才抓取），均不满足则跳过并**继续处理下一个链接**，直到最后一个链接处理完毕。

对每个链接执行：

1. **打开页面并判断帖子日期与评论数**

   - 使用 `mcp__chrome-devtools__navigate_page` 访问链接。
   - 等待页面初步加载（3-5 秒）。
   - **先判断帖子日期是否为近两日（今日或昨日）**：
     - **X/Twitter**：查找推文上的发布日期/时间（通常在推文顶部或在时间戳元素中），判断是否为今日或昨日。如果是近两日内则继续，否则直接跳过。
     - **YouTube**：查找视频发布时间（通常在标题下方，如 "XX hours ago"、"XX days ago"），判断是否为今日或昨日。如果是近两日内则继续，否则直接跳过。
     - 如果无法获取发布日期，则视为非近两日，直接跳过（不抓取非近两日内容的评论）。
   - **确认是近两日帖子后，再判断评论数**：
     - **X/Twitter**：查找评论数。必须使用 `[data-testid="reply"]` 元素的 `textContent` 或 `aria-label` 属性获取实际评论数（如 `aria-label="198 Replies. Reply"` → 198 条）。**注意：不要通过 DOM 中 `article[data-testid="tweet"]` 元素的数量来判断评论数**——页面只渲染了部分回复，实际评论数显示在回复按钮上。如果评论数 < 10，直接跳过，不抓取。
     - **YouTube**：查找评论总数（通常在评论区上方，如 "XX Comments"），如果评论数 < 10，直接跳过，不抓取。
     - 如果无法获取评论数，默认为低于10条，直接跳过。

2. **平台适配抓取策略（仅对评论数≥10条的帖子执行）**

   **X / Twitter：**

   - 页面加载后，多次向下滚动触发懒加载（每次滚动间隔 2 秒，至少滚动 5-8 次）。
   - 使用 `mcp__chrome-devtools__evaluate_script` 提取评论文本：
     - 选择器：`article[data-testid="tweet"]` 获取每条推文/回复
     - 跳过第一个（原推文），提取后续所有回复
     - 获取作者名（`[data-testid="User-Name"]`）和文本内容（`[data-testid="tweetText"]`）
   - 如果遇到登录墙、内容不可见或抓取失败，记录该链接状态并继续下一个。
   - **抓取完成后，立即继续处理下一个链接，不能因为已收集到足够评论就提前停止。必须处理完所有链接。**

   **YouTube：**

   - 页面加载后，先滚动到评论区（页面中下部分）。
   - 持续滚动评论区直到没有新评论加载（多次滚动，间隔 2-3 秒）。
   - 提取评论：
     - 选择器：`ytd-comment-thread-renderer` 或 `#content-text`
     - 获取作者名（`#author-text`）和评论内容（`#content-text`）
   - 过滤置顶评论和频道主自己的评论，优先保留真实用户评论。

3. **数据汇总**

   - 记录每个链接的处理结果：来源 URL、帖子日期是否近两日、评论数、是否抓取（近两日且≥10条才抓）、评论列表。
   - 如果某个链接抓取失败（如需要登录、页面结构变化等），记录失败原因，**继续处理下一个链接**。
   - **必须处理完阶段 1 校验通过的所有链接，不能跳过、省略或中途喊停。**
   - 最终汇总所有成功抓取的评论（**注意：仅包含近两日发布的帖子**），用于阶段 3 分析。

### 阶段 3：分析与输出（中文，直接输出到对话）

**进入本阶段前必须确认**：阶段 2 已遍历完所有链接，无遗漏。如果发现自己中途停止了遍历，必须先回去完成所有链接的处理再进入分析。

基于**所有成功抓取的评论**，生成五段式分析报告，直接在对话中输出，**不需要保存为文件**。

#### 第一部分：话题归纳
- 通读所有评论，识别出讨论的核心话题（如 "AI 编程工具效率"、"模型幻觉问题"、"AI 产品定价" 等）。
- 按话题归类，每个话题下列出相关评论的摘要（注明来源平台和原帖）。
- 格式：
  ```
  ## 第一部分：评论话题归纳
  
  ### 话题 1：XXXX
  - 来源：<平台> | <原帖标题/摘要>
  - 评论摘要：...
  
  ### 话题 2：XXXX
  ...
  ```

- **在话题归纳末尾，增加「金句捕获：陌生化视角」小节**：
  - 从所有评论中筛选出具有**陌生化、差异化、非人类视角**的评论——那种让你停下来、换个角度重新看问题的表述。
  - 陌生化视角的特征：不是抱怨，不是技术讨论，而是用一个意外的角度重新定义/描述一件事。比如"当你的表达变成了用于赚钱的时候，表达就变成了表演。"
  - 这种评论往往具有哲学意味、反直觉观察、或者用一句话重新定义了一个行业现象。
  - 每条金句保留原文 + 中文翻译 + 来源链接。
  - 格式：
    ```
    ### 🌟 金句捕获：陌生化视角
    
    1. **<一句话金句>**
       - 原始表达："<原文>"
       - 中文翻译："<中文翻译>"（如原文为英文）
       - 来源：<平台> | <链接>
       - 陌生化在哪：<简要说明这句话为什么让人停下来思考>
    
    2. ...
    ```

#### 第二部分：痛点提取
- 从评论中筛选出明确表达"遇到什么问题"的内容。
- 痛点特征：某人（或某类用户）在某场景/使用某工具时，遭遇了具体困难、阻碍或负面体验。
- 每条痛点必须保留原始引用（或高度忠实的改写）和来源链接。
- 如果原始表达是英文，必须提供中文翻译；如果原始表达已经是中文，则无需重复翻译。
- 格式：
  ```
  ## 第二部分：痛点提取
  
  1. **<痛点一句话概括>**
     - 原始表达："<原文>"
     - 中文翻译："<中文翻译>"（如果原文为英文，此项必填）
     - 来源：<平台> | <链接>
     - 涉及工具/场景：<如 Cursor、Claude、API 集成等>
  
  2. ...
  ```

#### 第三部分：痛点分类
- 将第二部分的痛点按性质分类，例如：
  - **性能/效率类**（响应慢、卡顿、费 token）
  - **准确性/质量类**（幻觉、理解错误、输出不稳定）
  - **成本/定价类**（价格贵、免费额度少）
  - **集成/生态类**（与现有工作流不兼容、API 难用）
  - **学习/上手类**（文档缺失、学习曲线陡峭）
  - **安全/风险类**（供应链攻击、数据泄露）
  - **就业/社会类**（岗位替代、招聘冻结）
  - **其他**
- 每个分类下汇总对应痛点编号及简述。
- 格式：
  ```
  ## 第三部分：痛点分类整理
  
  ### 1. 性能/效率类
  - 痛点 #1：...
  - 痛点 #5：...
  
  ### 2. 准确性/质量类
  ...
  ```

#### 第四部分：市场需求挖掘（产品开发方向）
- 通读所有评论，从中筛选出具有**市场价值的需求讨论点**，用于产品开发方向判断。
- 重点关注：
  - 用户明确表示"想要/需要/希望有这个功能"的表述
  - 用户因为现有方案不够好而在寻找替代品的表述
  - 用户自发提出的解决方案或变通方法（workaround），说明市场空白
  - 用户对价格敏感度的讨论
  - 用户在不同工具之间切换/对比的理由
  - 某一类用户群体的特殊需求（开发者、设计师、创业者、企业等）
- 每条需求必须有来源链接可追溯，必须是评论中真实存在的表述，不能编造。
- 如果原始表达是英文，必须提供中文翻译。
- 格式：
  ```
  ## 第四部分：市场需求挖掘
  
  ### 方向 1：<产品方向名称>
  
  1. **<一句话描述该需求>**
     - 原始表达："<原文>"
     - 中文翻译："<中文翻译>"（如果原文为英文）
     - 来源：<平台> | <链接>
     - 目标用户群：<如：前端开发者、AI 工具使用者、企业安全团队等>
     - 市场信号强度：🔴高 / 🟡中 / 🟢低（根据讨论热度和用户迫切程度判断）
  
  2. ...
  
  ### 方向 2：<产品方向名称>
  ...
  ```

#### 第五部分：文章写作灵感
- 从评论中提取可用于写文章的讨论点和切入角度。
- 输出格式为：一句话写出文章角度 + 简要说明为什么这个角度好 + 两条切入建议。
- 写文章角度时优先考虑以下方向（可以组合使用）：
  - **非人类视角**：一件工具、一只动物、一种植物、一个机构在自述。例如"一个 AI 代码审查插件的自述"、"一个 token 的自白"。
  - **具体的现代职业和场景**：理赔员、电梯保养工、菜市场摊主、夜班护士、外卖站长、二手房中介、分拣员……例如"当一个前端设计师开始用 AI 写代码"、"一个安全工程师眼中的 AI 插件市场"。
  - **微观尺度**：一次交易、一次门诊、一通电话、一次拆卸，内部发生的事。例如"一次 AI 代码审查的 30 秒"、"一次 API 调用从发出到限流的完整旅程"。
- 每篇文章角度需附来源链接，必须是评论中真实存在的素材。
- 格式：
  ```
  ## 第五部分：文章写作灵感
  
  ### 角度 1：<文章标题/角度>
  - **一句话角度**：<用一句话写出文章切入点>
  - **为什么这个角度好**：<说明为什么这个角度有传播力、有共鸣>
  - **切入建议**：
    1. <具体切入方式 1>
    2. <具体切入方式 2>
  - **素材来源**：<平台> | <链接> | 评论引用（摘取最有代表性的 1-2 条）
  
  ### 角度 2：<文章标题/角度>
  ...
  ```
  在五段式分析完成后，添加以下统计信息：

```
---

## 📊 抓取统计
- 总链接数：X 个
- 符合条件（近两日发布且评论数≥10条）并抓取的帖子数：X 个
- 非近两日发布跳过的帖子数：X 个
- 评论数<10条跳过的帖子数：X 个
- 抓取失败帖子数：X 个（如有）
- 总评论数：约 XXX 条
- 来源平台：X/Twitter（X 个）、YouTube（X 个）

## 📁 已保存文件
- `follow-builders-<日期>.md` — 当日 AI builders 动态（中文版，仅当日数据）
- `aihot-<日期>.md` — 当日 AI 热点资讯（中文版，仅当日数据）
```

## 注意事项

1. **前置条件检查**：阶段 1 的链接校验是强制步骤，绝不能在无链接的情况下进入阶段 2。
2. **必须遍历所有链接，不得中途停止**：阶段 2 必须处理阶段 1 校验通过的全部链接，不允许因为"已收集到足够评论"或"感觉数据已充足"而提前停止进入阶段 3。如果链接较多（如 >20 个），优先使用后台并行或批量处理方式加速，但绝不能跳过。
2. **日期与评论数双重筛选**：打开帖子页面后，**先判断发布日期是否为近两日（今日或昨日）**（非近两日直接跳过），**再判断评论数**（≥10条才抓取，<10条跳过）。两个条件必须同时满足。
3. **当日数据限制**：follow-builders 和 aihot 必须只获取今日的数据。
4. **中文版要求**：follow-builders 和 aihot 的 Markdown 输出必须是中文，每个帖子必须有摘要，不需要互动数据。
5. **浏览器状态**：假设 Chrome DevTools MCP 连接的浏览器已登录 X/Twitter 和 YouTube，如遇登录态失效导致抓取为空，需在输出中说明。
6. **评论去重**：同一用户在同一帖子下的相似评论只保留一条最完整的。
7. **语言**：所有分析报告必须使用中文输出。
8. **文件输出规则**：
   - **必须输出文件**：`follow-builders-<日期>.md`（中文版，带摘要，仅当日数据）、`aihot-<日期>.md`（中文版，带摘要，仅当日数据）
   - **无需输出文件**：评论分析的三段式报告直接在对话中展示
9. **X/Twitter 评论数判断要点（易错点）**：
   - **错误的做法**：通过 `document.querySelectorAll('article[data-testid="tweet"]').length - 1` 数 DOM 中渲染的回复元素。这只能拿到页面懒加载出的部分回复，不是真实评论数。
   - **正确的做法**：使用 `document.querySelector('[data-testid="reply"]')` 获取回复按钮元素，取其 `textContent` 或 `aria-label` 属性（如 `"198 Replies. Reply"` → 198条）。这是 X/Twitter 页面直接显示的真实评论总数。
   - **验证方法**：可以先取 `textContent` 确认数字，再取 `aria-label` 验证（如 `"198 Replies. Reply"` 格式）。两个值应该一致。
